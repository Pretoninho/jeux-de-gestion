# Mémoire du projet

Ce fichier résume tout ce qui a été discuté et décidé jusqu'ici, pour reprendre le travail sans tout re-dériver. À tenir à jour à chaque décision importante.

## Le projet

Jeu de gestion solo, gratuit, déployé sur GitHub Pages (frontend uniquement, pas de backend).

**Thème : médiéval-fantastique.** Historique des revirements, du plus récent au plus ancien : urbain/proche-futur (RPG Urban Kit — abandonné à son tour) → **médiéval-fantastique**, déclenché par l'utilisateur apportant le pack "Tiny Swords (Free Pack)" de Pixel Frog (téléchargé depuis itch.io, licence vérifiée directement sur la page : usage perso + commercial libre, modification autorisée, crédit apprécié non obligatoire, pas de redistribution des fichiers bruts). Avant lui : sci-fi (trop peu d'assets fiables trouvés, cf. section Assets) → pivot vers urbain/proche-futur pour exploiter à fond RPG Urban Kit (riche, vérifié, CC0) plutôt que d'assembler des packs sci-fi plus faibles. Le pack de contenu s'appelle maintenant `medieval` (`src/content/medieval/`) ; deux angles thématiques (artisanat, garnison) implémentés, voir section "Concept de jeu actuel". Contrairement à RPG Urban Kit, Tiny Swords fournit de vraies icônes de bâtiment entier par type (Château, Caserne, Monastère, Tour, Archerie, Maisons) — ça résout directement le problème "les icônes de bâtiment ne ressemblent pas à des bâtiments" identifié sous l'ancien thème (voir section Assets), sans avoir besoin de composer des sprites.

**Structure du jeu : à une seule échelle, PAS de paliers empilés.** Revirement majeur, acté et implémenté : la structure à N paliers d'échelle croissante du document de passation (`docs/jeu_gestion_paliers_passation.md`) a été **abandonnée** — l'utilisateur veut "un jeu simple qui ne scale pas". Remplacée par une économie à une seule échelle : bâtiments avec recette + capacité, stocks de ressources, vente automatique des biens finis, réinvestissement dans la capacité (voir section "Décisions d'architecture actées").

## Décisions d'architecture actées

- **Stack** : TypeScript + Vite + Vitest.
- **Séparation stricte** : `src/engine/` (simulation, zéro dépendance UI/thème) / `src/presentation/` (rendu générique) / `src/content/<theme>/` (données spécifiques à un thème, y compris ses assets).
- **Moteur agnostique au thème** : graphe de recettes libre (pas de patron d'inputs/outputs imposé), **économie à une seule échelle** (pas de paliers — abandonnés, voir plus bas).
- **Placement spatial** : chaque pack définit des `BuildingType` (recette + capacité + coût de construction + `footprint?: {width,height}` optionnel, défaut 1x1) et une `grid: { width, height }`. `build()` (`src/engine/simulation.ts`) place une instance dont `(x,y)` est le coin haut-gauche de son empreinte (vérifie que **tout le rectangle** est dans les limites + libre + argent suffisant, via `rectsOverlap()` contre l'empreinte de chaque bâtiment déjà posé) ; `EconomyState.placedBuildings` liste les instances posées (id, type, x, y) — la taille n'est pas dupliquée dans l'instance, elle se déduit du type via `footprintOf()`. `buildingAt(pack, state, x, y)` trouve le bâtiment (s'il y en a un) dont l'empreinte couvre une case donnée, pas seulement son origine. **Empreintes variables IMPLÉMENTÉES** (ex. `garrison-fortress` en 3x3, `src/content/medieval/index.ts`) — reste volontairement non fait : caméra/scroll, démolition/déplacement, aperçu de la zone constructible au survol avant de cliquer (le clic échoue silencieusement si l'empreinte ne rentre pas, même limitation UX que l'existant sur case occupée/argent insuffisant).
- `tick()` (`src/engine/simulation.ts`) : chaque bâtiment **placé** produit selon le stock disponible — `unités = min(capacité, min sur chaque input de stock/quantité)`. Les ressources avec `sellPrice` sont auto-vendues en fin de tick (stock → argent, remis à zéro). Ordre de `placedBuildings` (= ordre de placement) = ordre de consommation : un bâtiment placé plus tôt peut affamer un bâtiment placé plus tard compétant pour la même ressource — limitation connue et acceptée, pas un bug.
- Testé dans `simulation.test.ts` (10 tests) : `build()` (place/case occupée/hors limites/pas assez d'argent/type inconnu), `tick()` (production sans input, throttling par stock, un bâtiment qui en affame un autre, vente automatique, pack immuable).
- `GameLoop` (`src/engine/gameLoop.ts`) : temps réel avec pause, vitesses x1/x2/x5. Inchangé par les revirements successifs.
- **Rendu** : grille top-down simple (isométrique explicitement écarté — trop coûteux en solo). Tuiles affichées à **32×32 px**, upscale net via `image-rendering: pixelated`. Depuis le pivot vers Tiny Swords, chaque type de bâtiment a sa **propre icône de bâtiment dédiée** (pas un stand-in ressource) — voir section Assets pour l'historique de ce problème sous l'ancien thème.
- **Pipeline d'assets** (`src/presentation/assets.ts` + `tile.ts`) : type `ThemeAssets`/`SpriteRef`, fonction `renderTile()` avec **fallback placeholder coloré déterministe** (même id = même couleur) quand aucun sprite n'est mappé. `CREDITS.md` obligatoire pour toute source externe (auteur, lien, licence).
- **Sol de la grille (IMPLÉMENTÉ)** : `ThemeAssets` a un champ `ground?: SpriteRef` (pas indexé par id d'entité, une seule valeur pour toute la grille) peint comme `background-image` CSS de chaque `.grid-cell` ; le bâtiment posé, s'il y en a un, se dessine par-dessus via l'élément `.tile` habituel. `spriteToCss()` (extrait de `renderTile()`) mutualise le calcul CSS multi-calques entre les deux usages. Pack `medieval` : un seul tile d'herbe (rogné depuis `Terrain/Tileset/Tilemap_color1.png` de Tiny Swords, zone intérieure sans bordure d'autotile) réutilisé sur toute la grille — pas de variation de terrain pour l'instant, cohérent avec le scope v1 modeste ; variation possible plus tard sans changer le moteur.
- **La grille se met à jour par re-rendu complet, pas par diff incrémental** : `build()` retourne un résultat, et en cas de succès `main.ts` appelle `render()` qui vide `#grid` (`innerHTML = ''`) et reconstruit les 64 cases depuis `state.placedBuildings`. Aucune mécanique séparée à ajouter pour qu'un nouvel élément posé apparaisse — c'est déjà automatique à ce stade (petite grille fixe, coût négligeable). À revisiter uniquement si la grille grandit beaucoup (grille scrollable prévue plus tard, voir "Prochaines étapes").
- **Déploiement** : GitHub Actions (`.github/workflows/deploy.yml`) → build + test → GitHub Pages, sur push vers `main` ; site public : `https://pretoninho.github.io/jeux-de-gestion/` (testable directement, y compris sur mobile — pas besoin d'un dev serveur local).
- **Présentation/UX du harnais de dev (IMPLÉMENTÉ)** : passe de polish demandée explicitement par l'utilisateur ("développer le gameplay... présentation/UX" choisi parmi plusieurs priorités proposées), pour que le harnais cesse de se présenter comme un outil brut. Titre/pitch en tête de page ("Le Royaume"), fond parchemin + sections encadrées façon panneau en bois, tout texte faisant référence à `CLAUDE.md`/"v1 volontairement modeste" retiré du texte visible par le joueur (déplacé dans les commentaires code). **Boutons et jauges de satisfaction reskinnés avec de vrais éléments d'UI de Tiny Swords** (`tools/asset-composer/tiles/tiny-swords/ui-elements-*`) plutôt que du CSS générique :
  - Les feuilles de boutons/barres de la bibliothèque sont des grilles 64px avec des colonnes/lignes de gouttière vides (ex. bouton bleu = grille 5×5, seules les cases (0,2,4)×(0,2,4) sont utiles) — script Python ponctuel (`PIL`) pour recomposer ces 9 cases utiles en une image 3×3 sans gouttière, exploitable proprement en CSS `border-image`. Résultat stocké dans `src/assets/themes/medieval/tiny-swords/ui/` (`button-regular.png`, `button-pressed.png`, `bar-base.png` en 3-slice horizontal, `bar-fill.png`).
  - Boutons : `border-image` (state `:active` bascule vers la texture "pressed" pour un retour tactile).
  - Jauges de satisfaction : cadre en bois via `border-image` sur `.gauge__track`, remplissage `.gauge__fill` coloré par teinte (`toneFor()`, seuils repris de la maquette : ≥60 bon/≥35 avertissement/sinon danger) via `filter: hue-rotate()` sur la texture de remplissage plutôt que 3 textures séparées — un seul filtre "warn" (hue-rotate 25°) suffisamment décalé du "danger" (pas de filtre) pour rester lisible, "good" en hue-rotate 75° (vert olive, pas un vert franc à cause de la teinte de base rougeâtre du sprite).
  - La bannière parchemin (`ui-banners-*`) a été essayée mais écartée : feuille 704×512 sans grille régulière de 64px, découpage fiable pas possible sans deviner — le fichier n'a pas été gardé dans le projet.

## Concept de jeu actuel : économie à une seule échelle

Remplace la structure à paliers du document de passation (voir section suivante). `BuildingType` = recette (graphe libre d'inputs/outputs) + capacité + coût de construction ; le joueur **place** des instances sur une grille via `build()` (voir "Mode de contrôle" ci-dessous) ; stocks de ressources ; ressources marquées `sellPrice` vendues automatiquement chaque tick. La profondeur vient d'ajouter des recettes/types de bâtiments à ce même niveau, pas de changer d'échelle.

Pack `medieval` (`src/content/medieval/`, ex-`urban`, remplacé lors du pivot de thème), deux angles implémentés côte à côte, chacun préfixé et totalement indépendant de l'autre — miroir direct de la structure `urban` précédente (mêmes chiffres de capacité/coût, juste reskinnés) :
- **Artisanat** (`craft-`, faction Bleue) : Bois/Minerai d'or (bruts) → Planches/Lingots d'or (intermédiaires) → Équipement (bien vendu). 5 types de bâtiments : Bûcheron, Mine d'or, Scierie, Fonderie, Forge. Capacités volontairement déséquilibrées (Forge capacité 6 mais Fonderie ne fournit que 3 lingots/tick) pour garder vivant l'arbitrage capacité/flux.
- **Garnison** (`garrison-`, faction Rouge) : Viande (brut) → Provisions (intermédiaire) → Ravitaillement (vendu). 3 types de bâtiments : Pâturage, Auberge, Avant-poste. Chaîne volontairement plus courte que l'artisanat (1 seul brut, pas de convergence à 2 branches) — preuve que les angles n'ont pas besoin de la même profondeur.

Les deux angles tournent en parallèle dans le même `tick()`, chacun avec son propre argent cumulé (`state.money` est partagé, alimenté par les deux ventes). Grille du pack : 8×8, argent de départ 150 dans le harnais de dev (`src/main.ts`).

**Système budget/satisfaction — IMPLÉMENTÉ.** Objectif : créer une tension de gestion transversale (inspirations citées par l'utilisateur : le panneau budget de Cities Skylines, le bonheur de Pharaon/Cities, le délai de la crue du Nil — pris comme exemples du *principe* à reproduire, pas comme mécaniques à copier littéralement) sans ajouter un palier structurel : c'est une couche par-dessus l'économie à une seule échelle, pas une nouvelle échelle imbriquée.
- **Concepts moteur** (`src/engine/types.ts`, `simulation.ts`) : `BuildingType.sector?`/`Resource.sector?` (tag optionnel, ex. `'craft'`/`'garrison'` — un pack qui ne tague rien reste inerte face au système) ; `ContentPack.sectors?: SectorConfig[]` (`id`, `label`, `taxSensitivity?`) ; `ContentPack.budgetCategories?: BudgetCategory[]` (`id`, `label`, `weightBySector: Record<string, number>`, `costPerBuilding?`) — liste **extensible à l'exécution** via `addBudgetCategory(state, category)`, pas figée au pack (demande explicite de l'utilisateur : "qu'on puisse ajouter des catégories au fur et à mesure").
- **État** : `EconomyState.taxRate` (0-100, `setTaxRate()`), `.budgetCategories` (chaque entrée a un `level` 0-100 réglable via `setBudgetCategoryLevel()`), `.satisfactionBySector` (0-100 par secteur).
- **Formule (`tick()`)** : à chaque tick, `cible = 50 − (taxRate/100)×40×taxSensitivity + Σ_catégories ((level−50)/50)×25×poids_secteur`, bornée [0,100] ; la jauge de satisfaction avance vers cette cible de 12% de l'écart par tick (**inertie** demandée explicitement — une décision fiscale met plusieurs ticks à faire mal, et autant à se réparer, plutôt que de sauter instantanément). `multipliersForSatisfaction()` en dérive deux multiplicateurs (capacité de production ET prix de vente — "les deux" confirmé par l'utilisateur), appliqués respectivement à `type.capacity` des bâtiments du secteur et au `sellPrice` des ressources du secteur lors de l'auto-vente.
- **Argent** : les impôts encaissent un bonus direct (`revenue = ventes × (1 + taxRate/100)`) — monter les impôts rapporte tout de suite mais pèse sur la satisfaction, indépendamment de ce qu'elle finance. Chaque catégorie de dépense coûte `(level/100) × costPerBuilding × nombre_total_de_bâtiments_posés` par tick — **proportionnel à la taille de l'économie** (confirmé explicitement), donc le même curseur coûte plus cher à mesure que le joueur s'étend.
- **Différenciation par secteur** (confirmée explicitement) : chaque secteur a sa propre `taxSensitivity` et chaque catégorie a des poids différents par secteur — ex. pack `medieval` : `craft` (taxSensitivity 1.15) plus sensible aux impôts que `garrison` (0.85) ; catégorie "Écoles" profite plus à la garnison (0.65) qu'à l'artisanat (0.35), "Santé" l'inverse.
- **Testé** dans `simulation.test.ts` (10 tests dédiés) : seed à 50 par défaut, dérive progressive (pas de saut), différenciation de sensibilité fiscale entre secteurs, clamping de `setTaxRate`/`setBudgetCategoryLevel`, rejet d'un id de catégorie dupliqué/inconnu, sous/sur-financement qui tire la satisfaction en dessous/au-dessus de 50, throttling de la capacité par la satisfaction, réduction du prix de vente par la satisfaction (isolée de l'effet direct des impôts sur le chiffre d'affaires — sinon le bonus fiscal peut masquer la baisse de prix dans un test), coût proportionnel au nombre de bâtiments.
- **UI dans le harnais de dev** (`src/main.ts`) : section "Budget municipal" — curseur impôts, liste de catégories (chacune avec son curseur), bouton "+ Catégorie" qui demande nom + poids par secteur via `prompt()` (acceptable pour un harnais de dev, pas la vraie UI) et appelle `addBudgetCategory()` + `renderCategories()`. Jauges de satisfaction par secteur avec barre de progression + multiplicateurs affichés. **Piège évité** : les lignes de catégories/jauges ne sont reconstruites que sur changement structurel (init, ajout de catégorie) — jamais à chaque tick, sinon reconstruire le DOM d'un `<input type="range">` interromprait un glisser-déposer en cours ; seules les valeurs affichées (texte, largeur de barre) sont mises à jour à chaque tick via `updateGauges()`.
- **Maquette de validation** avant implémentation moteur (approche demandée par l'utilisateur : "maquette d'abord") : `tools/asset-composer/` n'est pas concerné, la maquette était un fichier HTML autonome publié en Artifact (hors-repo, jetable) reprenant les mêmes curseurs/jauges/inertie en JS vanilla pour valider l'interaction avant de coder le vrai moteur — validé par l'utilisateur ("Tant que c'est couplé comme il faut au jeu ensuite c'est ok").

**Mode de contrôle : grille spatiale avec placement — IMPLÉMENTÉ (v1 modeste).** Historique : d'abord tranché "dashboard, pas de grille" (raisonnement : éviter le gros chantier caméra/collisions, cohérent avec "jeu simple qui ne scale pas"). En expliquant ce choix, la distinction n'a pas été assez claire : "dashboard" a été présenté comme "pas de carte à la Pharaoh" sans dire explicitement que ça supprimait *toute* mécanique de construction (même un simple bouton "construire" sans carte). L'utilisateur pensait donc que la construction/placement restait possible en mode dashboard — ce n'était pas le cas, d'où le revirement une fois le malentendu clarifié.

**Implémenté** : le joueur sélectionne un type de bâtiment dans une palette (`src/main.ts`), clique une case vide de la grille pour le construire (`build()`). Rendu en `<div>` CSS grid, chaque case affiche `renderTile()` (sprite ou placeholder). Testé de bout en bout en interactif via Playwright (sélection palette → clic case → production/vente réelles observées).

**Écran d'arrivée + grille pleine taille d'écran — IMPLÉMENTÉ.** Deux tâches demandées explicitement par l'utilisateur ("concentre-toi juste sur ces 2 tâches", reste volontairement pas touché — présentation des autres sections/boutons hors scope de cette passe) :
- **Écran d'arrivée** : `renderLanding()` affiche titre + accroche + bouton "Nouvelle partie" avant toute autre chose ; le clic appelle `startNewGame()` qui crée un `EconomyState` frais (`createInitialState()`) et construit toute l'UI de jeu. `state`/`selectedType`/`lastProduced` sont maintenant des variables **locales à `startNewGame()`** (plus des `const`/`let` de module) — nécessaire pour qu'une future re-partie reparte d'un état propre plutôt que de réutiliser l'ancien. `loop` (le `GameLoop`) est recréé à chaque appel et l'ancien est mis en pause avant, pour éviter qu'une boucle abandonnée continue de tourner en arrière-plan.
- **Grille dimensionnée à l'écran (v1, carrée) — DÉPASSÉ, voir ci-dessous** : `updateGridSize()` calculait une taille de case = `min(largeur_fenêtre, hauteur_fenêtre) × 0.94 / dimension_max_grille`. Suffisant pour une grille carrée, mais laissait une bande de parchemin visible sur l'axe non contraignant (ex. en hauteur sur un téléphone portrait) — pas "toute la surface de l'écran" comme redemandé ensuite par l'utilisateur.
- **Grille plein écran + bordure d'eau (IMPLÉMENTÉ, 2e passe)** : demande reformulée explicitement — "affiche la map sur toute une surface d'un écran de téléphone", en s'appuyant sur le [guide officiel de tilemap Tiny Swords](https://pixelfrog-assets.itch.io/tiny-swords/devlog/1138989/tilemap-guide) (ordre de calques recommandé : BG Color → Water Foam → Flat Ground, la mousse se plaçant sur toutes les cases qui touchent l'eau). Grille du pack changée de carrée (8×8) à **rectangulaire** (`{width:6, height:12}`, `src/content/medieval/index.ts`) pour se rapprocher du ratio portrait d'un téléphone une fois bordée. `updateGridSize()` calcule la taille de case à partir des **deux dimensions de la fenêtre indépendamment** (`min(largeur/colonnes_totales, hauteur/lignes_totales)`) — nécessaire dès qu'une grille n'est plus carrée. `.grid-section` sort de la largeur/padding du `body` (technique `width:100vw; margin: calc(50% - 50vw)`) pour remplir tout le viewport horizontalement, posée en premier dans le DOM (avant l'en-tête) pour aussi remplir verticalement (`min-height:100vh`). Fond de la page = couleur d'eau plate (`--water: #47aba9`, échantillonnée sur `Water Background color.png`) au lieu du parchemin, pour que le moindre écart d'arrondi de taille de case se voie comme "plus de mer", jamais comme un vide d'interface.
- **Carte terrain éditable — IMPLÉMENTÉ (3e passe, remplace le premier jet)** : demande suivante de l'utilisateur — "créé outil pour que je créé moi même la map". Le rendu du sol n'est plus un anneau d'1 case codé en dur (`WATER_BORDER`) mais une vraie **carte de terrain peinte case par case**, `TerrainMap` (`src/presentation/terrain.ts` — `width`/`height` du canevas total, `buildableOffsetX`/`buildableOffsetY` = où se pose le rectangle constructible dedans, `tiles: string[][]` = un id de sprite terrain par case). Données concrètes dans `src/content/medieval/terrain.ts` (générées, pas censées être éditées à la main). **Toujours purement présentationnel** : `pack.grid` reste la seule zone que `build()` valide ; `main.ts` calcule les coordonnées "terrain" (`x = worldX − buildableOffsetX`) pour décider quoi rendre cliquable, exactement comme l'ancien `WATER_BORDER` mais piloté par les données plutôt qu'une constante.
  - Les ids de tuile terrain (`terrain-grass`, `terrain-water`, `terrain-water-foam`) vivent dans `medievalThemeAssets.sprites` au même titre que n'importe quel sprite de ressource/bâtiment (préfixe `terrain-` par convention, pas de champ dédié `ground`/`waterBorder` sur `ThemeAssets` — retirés, un seul mécanisme de lookup pour tout). `terrain-water` réutilise `Water Background color.png` (couleur plate) importé cette fois comme vraie image (pas juste une variable CSS), pour que l'éditeur puisse en afficher un aperçu.
  - **Outil de dev créé pour ça : `tools/map-editor/`** (accès via `npm run dev` → `/tools/map-editor/index.html`, jamais expédié — exclu du build de production comme `tools/asset-composer/`). Découvre les ids de tuile disponibles dynamiquement depuis `medievalThemeAssets.sprites` (préfixe `terrain-`) plutôt que de les coder en dur, donc un futur ajout (rochers, buissons...) apparaît dans la palette sans toucher l'outil. Le rectangle constructible (`pack.grid`, positionné via `buildableOffsetX/Y`) s'affiche toujours en herbe et est **verrouillé** (non cliquable, juste un repère visuel) — seule la zone décorative autour se peint. Largeur/hauteur/décalage ajustables (redimensionner conserve la peinture existante là où les coordonnées restent valides). **Enregistrer** régénère entièrement `src/content/medieval/terrain.ts` via un plugin Vite dev-only (`tools/map-editor/save-plugin.ts`, même pattern `configureServer` que l'asset composer) — pas de sections à préserver ici (contrairement à `assets.ts`), donc pas de marqueurs, le fichier entier est regénéré à chaque sauvegarde. Vérifié de bout en bout : peindre une case → enregistrer → recharger le jeu → le nouveau tile s'affiche.
- Le reste de la mise en page (palette/réserves/budget, sous la grille, atteignable en scrollant) n'a pas été retravaillé — accepté explicitement comme hors scope de cette passe ("laisse le dev tool [`tools/asset-composer/`] de côté, je ne veux pas qu'il serve d'interface de jeu" — clarification que ce travail se fait dans `src/main.ts`, pas dans l'outil de composition d'assets).

**Reset explicite de l'affichage post-landing — IMPLÉMENTÉ (4e passe).** L'utilisateur a signalé une vraie friction ("j'ai du mal à te faire faire ce que je souhaite") et a posé deux règles de travail qui restent valables pour la suite :
1. **Toujours confirmer par écrit ce qui va être fait avant de toucher au code** dès qu'une demande touche à la structure/l'affichage — ne plus enchaîner directement sur l'implémentation même quand l'intention semble claire.
2. **"On va reconstruire petit à petit"** — accepté explicitement que le jeu redescende à un état minimal, plutôt que d'accumuler des couches de présentation non demandées.

Conséquence concrète : `startNewGame()` n'affiche plus **que** la grille plein écran (rien d'autre — plus de titre, barre de statut, boutons Démarrer/vitesse, palette, réserves, budget). Tout le HTML/JS correspondant a été retiré de `src/main.ts` (pas juste caché en CSS) — `EconomyState`, `GameLoop`, `build()`/`tick()`, budget/satisfaction ne sont plus du tout référencés dans le harnais pour l'instant ; la grille se peint uniquement depuis `TerrainMap` (`medievalTerrain`), sans notion de bâtiment posé. Fourni par l'utilisateur sous forme de zip après l'avoir trouvé lui-même ; visuellement inspecté (château, maison) avant confirmation — clairement médiéval-fantastique/RTS (tags itch.io : Fantasy, Medieval, Real time strategy), ce qui a motivé le pivot de thème plutôt que de forcer un usage urbain/proche-futur. 410 fichiers PNG au total, catalogue complet rapatrié dans `tools/asset-composer/tiles/tiny-swords/` (noms aplatis depuis l'arborescence source, ex. `buildings-blue-buildings-castle.png`), 15 fichiers effectivement utilisés dans `src/assets/themes/medieval/tiny-swords/` : 9 bâtiments (icône dédiée par type, voir pack `medieval`), 5 icônes de ressource (bois, minerai d'or, lingots, viande, un outil générique), 1 tuile de sol rognée depuis le tileset. `craft-planks` et `garrison-provisions` restent volontairement en placeholder, faute de sprite distinct qui convienne (même limitation assumée que sous l'ancien thème).

**Historique — RPG Urban Kit (Kenney, CC0)** : intégré pour le thème urbain/proche-futur, **retiré du pack de contenu actif** lors du pivot vers médiéval-fantastique (fichiers utilisés dans le jeu supprimés de `src/assets/themes/`). Le catalogue complet (486 tuiles) reste dans `tools/asset-composer/tiles/rpg-urban-kit/` comme bibliothèque dormante — pas supprimé, juste plus référencé par le pack actif, récupérable si un futur thème/angle urbain revient.

**Raison de l'abandon de la piste sci-fi** : aucun pack trouvé n'égalait la richesse/fiabilité de RPG Urban Kit (480+ sprites, licence CC0 confirmée, scène assemblée vérifiée) — le meilleur candidat sci-fi (Aske4) restait à confiance moyenne, et les autres étaient soit limités en variété (11 icônes), soit de licence incertaine, soit pas du tout 2D. Plutôt que d'assembler plusieurs packs sci-fi imparfaits, décision d'exploiter à fond le pack déjà intégré et vérifié.

**Erreur à ne pas répéter** : la première recommandation d'Aske4 s'appuyait sur un résumé texte de la page (WebFetch) sans avoir regardé l'image — l'utilisateur a eu raison de douter. Toujours télécharger et inspecter visuellement l'aperçu avant de garantir qu'un pack convient (perspective, style), plutôt que de faire confiance à un résumé.

**Icônes de bâtiment (historique, sous l'ancien thème urbain) — problème depuis RÉSOLU par le pivot vers Tiny Swords, pas par la composition.** Constat de l'utilisateur à l'époque : le sprite "Casse auto" (réutilise l'icône wrench de sa ressource produite) ne ressemblait pas à une casse auto. Recherché des packs dédiés "une icône = un bâtiment entier" — **aucun trouvé qui convienne pour l'urbain/proche-futur** :
| Piste | Verdict |
|---|---|
| Kenney — City Kit (Industrial) | Modèles 3D, pas des sprites 2D |
| Kenney — Sketch Town | Kit de composition par tuiles (+ isométrique), pas des icônes autonomes |
| Cozy Village Shop Buildings (MutterPixel Studio) | De vrais bâtiments en sprite unique mais **payant** et thème "cozy médiéval", à l'opposé de l'urbain/proche-futur |
| RPG Urban Kit lui-même (re-vérifié en détail) | Confirmé : aucune tuile ne représente un bâtiment complet, uniquement des éléments de façade (murs/portes/fenêtres séparés) — c'est un kit de composition, pas d'icônes |

**Constat général (toujours valable)** : les icônes "un sprite = un bâtiment reconnaissable" sont rares en gratuit ; la plupart des kits (dont RPG Urban Kit) composent les bâtiments à partir de plusieurs tuiles. Tiny Swords fait exception (voir plus haut) — c'est spécifiquement ce qui a rendu le pivot de thème attractif en plus de la simple préférence esthétique.

**Décision actée et IMPLÉMENTÉE à l'époque (composition par calques)** : option 2 retenue puis codée — composer une icône en empilant **N sprites** (nombre libre, pas figé à 2) via `ThemeAssets`/`renderTile()` étendus (`SpriteRef` accepte `kind: 'composite'` avec `layers: FlatSpriteRef[]`, peint en CSS multi-background). Exemple validé sous l'ancien thème : `logi-scrapyard` (Casse auto) recomposait mur + porte. **Ce mécanisme reste dans le moteur** (toujours utilisable, ex. pour un futur angle qui manquerait de sprite dédié) mais le pack `medieval` ne l'utilise pas actuellement — tous ses bâtiments ont une icône dédiée directe.

**Outil de dev créé pour ça : `tools/asset-composer/`** (accès via `npm run dev` → `/tools/asset-composer/index.html`), toujours en place et à jour après le pivot. Fonctionnalités :
- Parcourt les tuiles par "kit" (dossiers de `tools/asset-composer/tiles/<kit>/` — `rpg-urban-kit/` (486 tuiles, dormant) et `tiny-swords/` (410 fichiers, actif) cohabitent).
- **Import de nouveaux kits** directement depuis l'outil (nom + sélection de fichiers PNG) — demandé explicitement par l'utilisateur, pas juste une bibliothèque figée.
- Composition par calques en nombre libre, avec réordonnancement et aperçu en direct.
- Association à un id du pack (`Resource.id`/`BuildingType.id`, autocomplété depuis `medievalPack`).
- **Écrit directement dans `src/content/medieval/assets.ts`** — demandé explicitement par l'utilisateur plutôt qu'un simple copier-coller. Un point d'API Vite (`configureServer`, donc actif uniquement en `npm run dev`, jamais dans le site déployé) copie les fichiers utilisés dans `src/assets/themes/medieval/<kit>/` et régénère les imports + entrées `sprites` dans une section délimitée par des marqueurs (`// asset-composer:imports:start/end`, `// asset-composer:sprites:start/end`) — tout le reste du fichier reste intouché.
- Vérifié de bout en bout via Playwright sous l'ancien thème (composition mur+porte → assignation → enregistrement → fichier réécrit → jeu affiche la nouvelle icône) ; chemins mis à jour vers `medieval` lors du pivot, pas re-testé de bout en bout depuis (pas nécessaire, le pack actif n'en a pas eu besoin).

**Piège rencontré et à anticiper** : si l'id composé existe déjà comme entrée manuscrite ailleurs dans `assets.ts`, l'enregistrement crée une **clé dupliquée** dans l'objet `sprites` — TypeScript refuse de compiler (`TS1117`, erreur détectée à la compilation, pas silencieuse). Il faut retirer l'ancienne entrée à la main après usage de l'outil. Pas automatisé (risque trop élevé de casser du code écrit à la main en essayant de le parser/supprimer automatiquement) — accepté comme limitation connue.

## Limitations d'environnement connues

- Le navigateur headless (Chromium/Playwright) de cet environnement **ne peut atteindre aucun site HTTPS externe** à travers le proxy réseau imposé (`ERR_CONNECTION_CLOSED` systématique, y compris sur `example.com` — testé avec plusieurs configurations). `curl`, lui, fonctionne très bien à travers ce même proxy. Conséquence pratique : impossible d'automatiser un téléchargement itch.io par navigateur (le flux de paiement/téléchargement est piloté en JS) → **demander à l'utilisateur de télécharger et déposer le fichier lui-même** plutôt que d'essayer de contourner.
- Les captures d'écran du serveur de dev **local** (`localhost`) fonctionnent normalement (pas de proxy nécessaire en loopback). Process utilisé : `npm run dev -- --port 5173 --strictPort` en arrière-plan, puis script Playwright avec `executablePath` explicite vers `/opt/pw-browsers/chromium-1194/chrome-linux/chrome` et `NODE_PATH=/opt/node22/lib/node_modules` (Playwright est installé globalement, pas comme dépendance du projet).
